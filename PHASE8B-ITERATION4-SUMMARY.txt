================================================================================
ROOT CAUSE ANALYSIS
================================================================================

Why Does Min-P Underperform?

1. FILTERING MECHANISM
   v2.2 (top_p/top_k): Dual-constraint
   - Keep top_k=20 candidates
   - Among those, keep cumulative probability ≤ 0.8
   - Result: ~5-15 available tokens
   - Provides diverse options at each step

   v2.3 (min_p): Single-constraint
   - Keep only tokens where p ≥ 0.05 × p_max
   - Result: ~3-8 available tokens
   - More restrictive, fewer options

2. IMPACT ON MULTI-STEP REASONING
   Single-step (TC1-TC6): Both work
   - High-probability correct action available in both

   Multi-step (TC7-TC8): v2.2 succeeds, v2.3 fails
   - First step: Both work (high-probability action)
   - Second step: Model needs to "continue reasoning"
     * v2.2: More diverse candidates → model explores "read_file" option
     * v2.3: Fewer candidates → model exits with EOS token instead

3. DIVERSITY-PERFORMANCE TRADEOFF
   v2.2: Higher diversity → maintains context across steps
   v2.3: Lower diversity → good for single decisions, fails for sequences

================================================================================
CRITICAL FINDINGS
================================================================================

1. PARAMETER COMPLEXITY IS JUSTIFIED
   The three-parameter config in v2.2 (temperature + top_k + top_p) is
   empirically superior to simpler alternatives. Simplicity ≠ superiority.

2. MATHEMATICAL VS. EMPIRICAL OPTIMIZATION
   Min-p is mathematically simpler, but empirically worse.
   Test-driven results trump theoretical elegance.

3. MULTI-STEP WORKFLOWS ARE SENSITIVE TO SAMPLING
   The regression occurs specifically in multi-step scenarios.
   Single-step tasks are robust to parameter changes.

4. TOP_P AND TOP_K INTERACTION
   The combination of top_p (cumulative cutoff) and top_k (candidate limit)
   provides better control than min_p's single threshold.
